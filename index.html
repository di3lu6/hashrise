Penetration Testing
Internal Penetration Testing

Internal Penetration Testing

Description

It is a security test where we simulate a real attack from within the internal network (by an intruder or a disgrunted employee), in order to detect how vulnerable your company's network really is.

The Penetration Testing starts with the same tasks included in a Vulnerability Assessment. After that, the real risk level is verified by exploiting the found vulnerabilities and all their possible combinations.

Its phases are:

  • Discovery.
  • Vulnerability Analysis.
  • Exploitation.
  • Privilege escalation and mantaining access.

Wireless Penetration Testing

Description

Wireless netwoks are an extension of your company's perimeter, and can be used as a gateway to your internal infrastructure.

An unauthorized Access Point, installed by an employee, which doesn't follow the company's security guidelines can be used to compromise your security.

Application Security
Web Application Analysis

Web Application Analysis

Description

During the analysis of a Web Application, manual and automated tests are run in order to detect insecure implementations or vulnerabilities that might have been introduced by development mistakes. These vulnerabilities usually allow access to sensitive information, escalation of privilege within the application, and even Operating System access.

The analytical process used by Cinta Infinita is based on the OWASP and OSSTMM methodologies, and always includes checks for the vulnerabilities that compose the OWASP Top 10 list.

Because every application's logic is different and vastly dependant on bussiness rules, we also look for bussiness logic vulnerabilities that wouldn't be detected through automatic analysis.

Application Security
Source Code Audit

Source Code Audit

Description

The source code audit (or "white box" audit) helps you get a better understanding of the security level of the developed applications, by detecting vulnerabilities and security problems introduced during the stages of "Requirement Analysis", "Design" and "Development". Because the source code is available to the auditor, it is possible to detect vulnerabilities that wouldn't be detected with other approachs, eg. backdoors.

Every security component of the applications is deeply analyzed with the support of the development team. This allows us to get a better knowledge of the application and bussiness logic.

Special attention is always paid to the components that support critical functions like:

  • Authentication.
  • Authorization.
  • Input data validation.
  • Error and exception handling.
  • Operating System calls.
PCI-DSS Required Services
Penetration Testing (PCI)

Penetration Testing (PCI)

Description

The PCI-DSS (Payment Card Industry Data Security Standard) requires Penetration Testing to be conducted to ensure the security of the credit card data environment. Since PCI-DSS 3.0 all Merchants, Processors, Payment Gateways and Service Providers must conduct Penetration Testing tasks that:

  • Cover the entire cardholder data environment.
  • Cover both application-layer and network-layer threats.
  • Are based upon an industry-accepted model.
  • Cover both enternal and internal threats.

Penetration testing must be done on an annual basis and after any significant change in the infrastructure or applications.

Cinta Infinita's Penetration Testing services are aligned for PCI-DSS compliance.

Training/Courses
RFID/NFC Security Training

RFID/NFC Security Training

Description

This workshop will provide you with all the tools, materials, and references for further study and research. The workshop covers from the Low Frequency band (mainly used for individual physical access to buildings, garages, hotels, etc.) to the High Frequency band, where NFC is the main term we are going to discuss about.

We will understand which type of access cards can be emulated or even cloned; then we are going to use traditional NFC USB readers, we will compile and execute the famous LibNFC, and play with some special hardware like the Proxmark3 and HydraNFC.

Arduino examples will be shown to deal with both Low and High frequency cards and tokens. We will try to create our own cloner in less than 50 lines of code!

The last part of the workshop will review HID iClass vulnerabilities and focus on NXP Mifare Classic technology and public hacks.

By the end, we will discuss some case studies, practical hacks and lessons learned from working systems the can be found in the wild.


Duration: 8hs (1 full day or 2 half days)
